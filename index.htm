RMS Support Tool
Microsoft© Customer Service & Support EMEA

The 'RMS Support Tool' provides the functionality to reset all Microsoft® AIP/MIP/AD RMS client services. Its main purpose is to delete the currently downloaded policies and reset the user settings for AIP/MIP/AD RMS service, and it can also be used to collect data for troubleshooting.

Introduction

This information gives you background and usage information for the RMS_Support_Tool Microsoft® Windows® PowerShell script module.

Note:

  • Please only run RMS_Support_Tool if you have been prompted to do so by a Microsoft® support technician.
  • Do not modify or delete any component of the RMS_Support_Tool or the resulting trace files, as this will result in incorrect information when analyzing your environment.
  • It is recommended to test the RMS_Support_Tool with a test environment before executing it in a live environment.
  • There is no support for the RMS_Support_Tool. Please see the disclaimer below.
  • Nomenclature:
    • AIP = Azure Information Protection.
    • MSIP/MIP = Microsoft® Information Protection
    • MSIPC = Microsoft® Information Protection Client.
    • RMS = Rights Management Service.
    • AD RMS = Active Directory Rights Management Service.

Requirements

Before you can use the RMS_Support_Tool make sure that your environment fulfils the following requierements. Please update your environment if necessary.

Note:

  • The RMS_Support_Tool supports the following operating systems:
    • Microsoft® Windows® 8, Windows® 8.1, Windows® 10, Windows® Server 2012, Windows® Server 2012 R2, Windows® Server 2016 and Windows® Server 2019.
  • The RMS_Support_Tool supports the following Microsoft® Office versions:
    • Microsoft® Office 2010, Microsoft® Office 2013, Microsoft® Office 2016, Microsoft® Office 2019, and Microsoft® 365.

1. Microsoft® Azure Information Protection cmdlets (optional)

The Microsoft® Azure Information Protection cmdlets are installed with the Azure Information Protection client. Please ensure to have the latest version of the "AzureInformationProtection" cmdlets installed by checking its client version release history.

Currently, if you install the cmdlets as one user and run the cmdlets on the same computer as another user, you must first run the following command with administrative permissions in a Windows® PowerShell command window:

PS C:\> Import-Module -Name AzureInformationProtection

Note:

  • Microsoft® Azure Information Protection cmdlets are required to run the analyze option [T] Protection.

2. Internet access (required)

The RMS_Support_Tool consumes the following sources from the internet:

For logging and tracing purposes, the tool DebugView from Windows® Sysinternals is required. If you call the help file, the RMS_Support_Tool download its help file (RMS_Support_Tool .htm).

If you do not have the corresponding files on your local hard drive, the RMS_Support_Tool will automatically try to download the following files from the internet:

  • DebugView
  • RMS_Support_Tool.htm

Note:

  • Unexpected errors may occur, and some features may be limited, if there is no connection to the Internet.
  • If you’re using the RMS_Support_Tool in an environment that does not have internet access, you need to proceed with the manual installation.

3. Exchange Online PowerShell V2 module (recommended)

The Microsoft® Exchange Online PowerShell V2 module is required to proceed the option [L] Labels and policies.

If you do not have this module installed, the RMS_Support_Tool will install it from Powershell Gallery, but you must run a new instance of PowerShell for it to work after installation.

Currently, if you install the cmdlets as one user and run the cmdlets on the same computer as another user, you must first run the following command with administrative permissions in a Windows® PowerShell command window:

PS C:\> Import-Module -Name ExchangeOnlineManagement

Installation via PowerShell Gallery (recommended)

RMS_Support_Tool is available on PowerShell Gallery.

1. Install PowerShellGet

The PowerShellGet-module, and the NuGet provider are responsable for consuming the RMS_Support_Tool package from PowerShell Gallery repository.

Note:

  • For additional information you can also see the following information: Installing PowerShellGet.

1.1. Install NuGet provider

To install the latest NuGet provider you must run the following command:

PS C:\> Install-PackageProvider -Name NuGet -Scope CurrentUser -Force

1.2. Install PowerShellGet-module

Second, you must install the latest PowerShellGet module using this command:

PS C:\> Install-Module -Name PowerShellGet -AllowClobber -Scope CurrentUser -Force

1.3. Set PowerShell Gallery as a trusted repository

If PowerShell Gallery is not already a trusted repository in your environment, you must define and confirm it as such using the following command:

PS C:\> Set-PSRepository -Name PSGallery -InstallationPolicy Trusted

2. Install RMS_Support_Tool

To install the RMS_Support_Tool you must run the following command:

PS C:\> Install-Module -Name RMS_Support_Tool -Scope CurrentUser

3. Allow signed PowerShell scripts

If PowerShell script execution is restricted in your environment, you must bypass this restriction to run the RMS_Support_Tool. To do this, run the "Set-ExecutionPolicy" command with the following parameters:

PS C:\> Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned -Force

Know issues:

  • Set-ExecutionPolicy has no effect. This applies only to a manual installation: If you downloaded the RMS_Support_Tool from an untrusted website, it will be extended by an Alternate Data Streams in NTFS (ADS). If the corresponding website in the ADS is not trusted in your environment, the PowerShell command "Set-ExecutionPolicy" has no effect. There're two methods to solve this issue: Either you right-click the downloaded script file and "Unblock" it, or you add the correcponding website to your trusted sites settings in Internet Explorer.
  • Script does not start with the error: Set-ExecutionPolicy : PowerShell updated your local preference successfully, but the setting is overridden by the group policy applied to your system or Set-ExecutionPolicy : Windows PowerShell updated your execution policy successfully, but the setting is overridden by a policy defined at a more specific scope. If you see this message in a PowerShell command window, you are most likely affected by the Windows® Group Policy setting "Turn on Script Execution". To resolve this problem, you must remove this Group Policy setting or configure a setting that allows scripts to run. Please request assistance from your administrator to do this. Please also read Use Group Policy to Manage Execution Policy and Set-ExecutionPolicy.

4. Check installation

To verify if the installation was successful, you can call the "Get-Module" cmdlet with the following parameter:

PS C:\> Get-Module -ListAvailable

If you find an entry like the following, the installation was successful:

PS C:\> Get-Module -ListAvailable

 

    Directory: C:\Users\<UserName>\Documents\WindowsPowerShell\Modules

 

ModuleType   Version   Name                     ExportedCommands

----------   -------   ----                     ----------------

To verify that an installation was performed using PowerShell Gallery, you can run the following command:

PS C:\> Get-InstalledModule -Name RMS_Support_Tool

If you find an entry like the following, the installation was performed by PowerShell Gallery:

PS C:\> Get-InstalledModule -Name RMS_Support_Tool

 

Version     Name                 Repository     Description

-------     ----                 ----------     -----------

1.3.1       RMS_Support_Tool     PSGallery      The 'RMS Support Tool' provides the functionality to reset...

 

Manual Installation (not recommended)

1. RMS_Support_Tool

To install the RMS_Support_Tool manually, you must copy all its files (RMS_Support_Tool.psm1, RMS_Support_Tool.psd1, and RMS_Support_Tool.htm) to the following directory:

%UserProfile%\Documents\WindowsPowerShell\Modules\RMS_Support_Tool\1.3.1

Note:

  • This path must be listed in the PSModulePath environment variable. For more information, please read how to view the PSModulePath variable.
  • Please pay attention to point 3. Allow signed PowerShell scripts.
  • Unexpected errors may occur, and some features may be limited if there is no connection to the Internet.

2. Sysinternals DebugView

If you're working in an environment that does not have internet access, in addition you must copy all files of the DebugView tool from Windows® Sysinternals (Dbgview.exe, dbgview.chm, and Eula.txt) into the following directory:

%UserProfile%\Documents\WindowsPowerShell\Modules\RMS_Support_Tool\1.3.1\DebugView

Note:

  • To verify if the installation was successful, please review the check installation section.

 

Update

To check if a new version is available, you need to run the following command:

PS C:\> RMS_Support_Tool -CheckForUpdate

If a new version should be found, you must run the following command to update to the latest version:

PS C:\> Update-Module -Name RMS_Support_Tool -Force

Note:

  • Under certain circumstances, you may need to run the RMS_Support_Tool with administrative permissions to perform an update. Please request assistance from your administrator.
  • If the RMS_Support_Tool was not installed via PowerShell Gallery, any older version must first be removed before an update or installation can be performed.

 

Uninstall

In case you want to uninstall the RMS_Support_Tool, and it was installed via PowerShell Gallery, you must run the following command:

PS C:\> Uninstall-Module -Name RMS_Support_Tool

If you want to remove a previous version, for example version 1.3.0, you must run the following command:

PS C:\> Uninstall-Module -RequiredVersion "1.3.0" -Name RMS_Support_Tool

If the RMS_Support_Tool was installed manually, you must also remove it manually by deleting its installation folder.

Note:

  • Under certain circumstances, you may need to run the RMS_Support_Tool with administrative permissions to perform an uninstall. Please request assistance from your administrator.

 

User experience

To run the RMS_Support_Tool, click the Microsoft® Windows® START button and start typing "power". While you type, you'll see suggestion "Windows PowerShell". Click either "Windows PowerShell" or "Windows PowerShell ISE".

To start the RMS_Support_Tool , simply type "RMS_Support_Tool" into the PowerShell command window, and press enter. For example:

PS C:\> RMS_Support_Tool

When you start the RMS_Support_Tool, you'll see the following default menu:

  [I] INFORMATION

  [D] DISCLAIMER

  [H] HELP

  [R] RESET

  [P] RECORD PROBLEM

  [C] COLLECT

  [Y] ANALYZE

  [Z] COMPRESS LOGS

  [X] EXIT MENU

 

Please select an option and press enter:

IMPORTANT: For options [P] (RECORD PROBLEM), [A], and [L] (COLLECT) you must run the RMS_Support_Tool with administrative permissions. For option [U] (ANALYZE) you may run with with administrative permissions.
To do so, click the Microsoft® Windows® START button and start typing "power". While you type, you'll see suggestion "Windows PowerShell". Right-click this entry. In the context menu, select "Run as administrator".

You can also start the RMS_Support_Tool within the command-line. Use the following command-line parameter to see a short summary of all available command-line parameters:

PS C:\> RMS_Support_Tool -Information

To see a full list of all command-line parameters with description, run the following command:

PS C:\> Get-Help RMS_Support_Tool -Detailed

To see the help for a single command-line parameter, for example for "-CompressLogs", run the following command:

PS C:\> Get-Help RMS_Support_Tool -Parameter CompressLogs

 

Features

RMS_Support_Tool provides the following features (menu entry / command-line parameter):

[I] INFORMATION / -Information

This shows syntax and a description.

[D] DISCLAIMER / -Disclaimer

This displays the disclaimer of warranty.
Please read it carefully, and act accordingly.

[G] HELP / -Help

This opens the help file (RMS_Support_Tool.htm).

[R] RESET / -Reset <String>

IMPORTANT: Before you proceed with any of the following reset options, close all applications that run or use Microsoft® Office or AIP/MIP/AD RMS.

This option removes all AIP/MIP/AD RMS certificates, policy templates and all corresponding registry keys.

Note:

  • The default argument for reset via menu is "Client".
  • When an Office 2013 installation is detected, modern authentication (ADAL) is automatically enabled as a precaution.

Valid <String> arguments are: "Client", "Silent", or "Machine":

Client:

This argument removes all AIP/MIP/AD RMS certificates, policy templates and all corresponding user registry keys:

PS C:\> RMS_Support_Tool -Reset Client

This is the recommended argument to reset all Microsoft® AIP/MIP/AD RMS user client settings.

Silent:

This command line-parameter argument does the same as "-Reset Client", but does not print any output - unless an error occurs when attempting to reset:

PS C:\> RMS_Support_Tool -Reset Silent

If a silent reset triggers an error, you can use the additional parameter "-Verbose" to find out more about the cause of the error:

PS C:\> RMS_Support_Tool -Reset Silent -Verbose

You can also review the Script.log file for errors of silent reset.

Machine:

This argument does the same as "-Reset Client", but does also remove all AIP/MIP/AD RMS certificates, policy templates, custom configurations, and all corresponding machine registry keys:

PS C:\> RMS_Support_Tool -Reset Machine

Before, the RMS_Support_Tool creates a backup copy of existing custom configurations from the following registry key:

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSIPC\ServiceLocation]

The name of the backup file is ServiceLocationBackup.reg.

Note:

  • You must run the RMS_Support_Tool with administrative permissions to proceed this option. Please request assistance from your administrator.

[P] RECORD PROBLEM / -RecordProblem

IMPORTANT: Close all applications that run or use Microsoft® Office or AIP/MIP/AD RMS before you proceed with this option.

In a first step, this parameter reset silent the client, then it activates the required logging, tracing or debugging mechanisms by implementing registry settings, and enabling some Windows event logs. This process will be reflected by a progress bar “Enable logging...".

In a second step asks you to reproduce the problem. While you’re doing so, the RMS_Support_Tool collects and records data. Once you have reproduced the problem, all collected files will be stored into the default logs folder (%temp%\RMS_Support_Tool\Logs). Every time you call this option, a new unique subfolder will be created in the logs-folder that reflects the date and time when it was created, e.g. “180520-133005”. While the files are being cached, you will see a progress bar “Collecting logs...".

In the last step, the RMS_Support_Tool resets all activated log, trace, and debug settings to their defaults. This process will be reflected by a progress bar “Disable logging...".

You can then review the recorded log files in the logs folder.

Note:

  • You must run the RMS_Support_Tool with administrative permissions to proceed this option. Please request assistance from your administrator.
  • An additional tool from Windows® Sysinternals (DebugView) is required for logging. If you do not have this tool on your local hard drive, it will be automatically downloaded from the internet.

[U] Endpoint URLs / -AnalyzeEndpointURLs

This parameter analyzes important enpoint URLs.
The URLs are taken from your local registry or your tenant's AIP service configuration information, and extended by additional relevant URLs.

In a first step, this parameter is used to check whether you can access the URL.

In a second step, the issuer of the corresponding certificate of the URL is validated.

This process is represented by an output with the Tenant Id, Endpoint name, URL, Issuer, and the Status of the validation of the certificate issuer. For example:

-----------------------------------------------

Tenant Id: 48fc04bd-c84b-44ac-91b7-a4c5eefd5ac1

-----------------------------------------------

 

Endpoint: CertificationDistributionPointUrl

URL:      https://48fc04bd-c84b-44ac-91b7-a4c5eefd5ac1.rms.na.aadrm.com/_wmcs/certification

Issuer:   C=US, S=Washington, L=Redmond, O=Microsoft Corporation, CN=Microsoft Secure Server CA 2011

Status:   Ok (200)

In addition, analyze results are written into log file EndpointURLs.log in the subfolder "Analyze" of the Logs folder.

Note:

  • You must run the RMS_Support_Tool with administrative permissions to proceed this option, if the corresponding Microsoft® Office desktop applicaiton is not bootstraped. Please request assistance from your administrator.
  • You need to know your Microsoft® 365 global administrator account information to proceed with this option, as you will be asked for your credentials.

[T] Protection / -AnalyzeProtection

This option analyzes whether the current user is able to use protection.

Therefore an ad-hoc protection policy for custom permissions is created, and used to validate the protection with a sample file (Protection.txt). The result is represented by an output with the status of that process. For example:

License      : {[Users, RMSSupToolEncrTest@microsoft.com], [Permissons, VIEWER]}

File         : C:\Users\<UserName>\AppData\Local\Temp\RMS_Support_Tool\Logs\Analyze\Protection.ptxt

Verification : Successfull

In addition, analyze results are written to the Protection.log file, and the resulting protected file (Protection.ptxt) is stored in the subfolder "Analyze" of the Logs folder.

Note:

  • Please pay attention to point 1. Microsoft® Azure Information Protection cmdlets of the requirements section of this help file.

[Z] COMPRESS LOGS / -CompressLogs

IMPORTANT: This command-line parameter should always be used at the very end of a scenario.

This parameter compress all collected logs files and folders into a .zip file, and the corresponding path and file name is displayed. In addition, the default logs folder (%temp%\RMS_Support_Tool\Logs) is cleaned.

After this step you can send/upload the .zip file for the Microsoft® support technician.

Script log file

The RMS_Support_Tool creates the following log file. The log file give an overview about the commands that were run and offers a control mechanism to review the results.

Script.log

This log file collects the actions that has been taken and lists the results. It also logs general environment information from the used client machine, like the Windows® edition or Windows® PowerShell version.

 

Log files and folders

The RMS_Support_Tool collects the following log files and folders. If you want to learn more about the method used to collect a log or folder, please review the source code of the RMS_Support_Tool script module and read the comments.

Note:

  • Please note that not all log files are collected or recorded at all times. It depends on your environment which logs, or traces can be recorded.

AipServiceConfiguration.log

This file is only collected if you selected [A] AIP service configuration. The file contains the Azure Information Protection service configuration of your tenant.

For more information, please see the following documentation: Get-AipServiceConfiguration.

AipServiceTemplates.log

This file is only collected if you selected [A] AIP service configuration. The file contains the Azure Information Protection service template details.

This file contains a list of currently running processes on the local computer.

These files show all current TCP/IP network configuration settings.

This file is a network trace recorded by Netsh.exe.

This file contains debug output for all MSIPC- or MSDRM-clients on your local system. It was created by DebugView.exe. For more information, please see the following documentation: DebugView.
